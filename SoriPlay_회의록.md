## 회의 1 — 프로젝트 주제 선정

### 논의 배경
- 음성인식 수업 프로젝트 주제 선정 필요
- 대상: 우크라이나 난민 및 중앙아시아 고려인 초등학생
- 방향: 번역보다 **한국어 스피킹**에 중점

### 결정 사항
- 형태: **게임형 앱** (초등학생 집중력 및 동기부여)
- 핵심 기능: 발음 교정보다 **억양(인토네이션) 교정** 메인
- 차별점: 고려인 아동 특화 + Prosody 분석 + 게임형 태블릿 앱

### 교수님 자료 검토 (기존 팀 주제와 차별화)
기존 팀 주제 (피해야 할 것):
- Chrome add-on for YouTube
- Log-like defense game
- Honorific system
- Code-switching 분석
- Role playing based pronunciation correction
- Old fairy tales-based speech training
- Speech-along coaching
- Pronunciation correction tool
- Grammar Correction

**→ 고려인 아동 특화 + 억양 교정 + 게임 조합은 없음 → 차별화 확인**

## 회의 2 — 현장 답사 (안산 아리랑 고려인 지원 센터)

### 방문 일시
2025년 3월

### 주요 확인 사항

| 항목 | 내용 |
|---|---|
| 난민 규모 | 안산 거주 우크라이나·고려인 난민 약 3,000명 |
| 출신 | 카자흐스탄, 우즈베키스탄 등 중앙아시아 |
| 센터 현황 | 아동 26명 + 성인 35~40명 수업 중 |
| 학습 도구 | 교과서만 사용 (디지털 도구 전무) |
| 기기 사용 | 스마트폰 위주, TikTok 많이 사용 |

### 센터 대표 주요 발언

> *"특히 발음과 인토네이션이 잘 안 돼요. 러시아 억양에 맞춰서 한국어를 발음하다 보니까, 우리가 딱 들었을 때 바로 외국인이구나 라고 느낄 정도예요."*

> *"병원, 은행, 관공서, 출입국 사무소 — 이 세 가지만 해도 굉장히 좋은 아이디어인 것 같아요."*

> *"게임 형태 — 아이들이 굉장히 좋아할 것 같아요."*

### 수업 참관 추가 관찰
- 고려인끼리만 러시아어로 말함 → 한국어 발화 기회 전무
- 듣기는 되는데 말하기를 못 함 (수용 능력 > 발화 능력)
- 억양이 굳어지기 전인 아동기에 교정 필요

### 동음이의어 문제
센터 대표가 직접 언급:
- `쓰다` — 모자를 쓰다 / 우산을 쓰다 / 글을 쓰다
- `끼다` — 장갑을 끼다
- `피다` — 꽃이 피다 / 담배를 피다 / 바람을 피다
- 러시아어는 이런 구분 없이 단어 하나로 표현 → 매우 어려워함

## 회의 3 — 최종 주제 확정

### 최종 주제
**"고려인 아동을 위한 AI 한국어 말하기 연습 게임 앱 — SoriPlay"**

### 핵심 컨셉
> *"말할 기회가 없는 고려인 아동을 위해, AI가 한국어 말하기 상대가 되어주고 억양까지 교정해주는 게임 앱"*

### 핵심 기능 3가지

| 기능 | 설명 | 기술 |
|---|---|---|
| AI 대화 상대 | 아이가 말하면 AI가 자연스럽게 반응 | STT + LLM |
| 억양 피드백 | 러시아 억양 패턴 감지, 피치 곡선 시각화 | Prosody / Pitch 분석 |
| 실생활 롤플레이 | 병원·은행·출입국 시나리오 | 시나리오 스크립트 + STT |

## 회의 4 — 기술 스택 결정

### 전체 파이프라인
```
아이 발화 → STT → Prosody 분석 → AI 응답 생성 → 피드백 출력
```

### 기술 스택

| 영역 | 도구 | 비고 |
|---|---|---|
| 음성 인식 (STT) | OpenAI Whisper / Google STT | 한국어 정확도 비교 후 선택 |
| Prosody 분석 | librosa / Praat | 피치 추출, 러시아 억양 비교 |
| AI 대화 | Google Gemini API | 무료 티어 (1,500회/일) |
| 발음 유사도 | DTW / 코사인 유사도 | 속도 차이 보정 가능 |
| 앱 플랫폼 | Flutter (iOS/Android) | 태블릿+모바일, 오프라인 모드 |

### LLM 선택 이유 — Google Gemini
- GPT-4o, Claude API는 유료
- Gemini 무료 티어: 하루 1,500회 → 프로젝트 개발/테스트 충분
- 한국어 응답 품질 사용 목적에 적합

## 회의 5 — 백엔드 구조 설계

### 백엔드 폴더 구조
```
soriplay-backend/
├── main.py
├── routers/
│   ├── stt.py
│   ├── prosody.py
│   └── chat.py
├── services/

│   ├── prosody_service.py
│   └── gemini_service.py
├── models/
│   └── schemas.py
├── .env
└── requirements.txt
```

### 개발 순서 (중요)
1. **STT 먼저** → Whisper 연동, 한국어 인식 테스트
2. **Gemini 연동** → STT 텍스트를 AI에 넘기는 파이프라인
3. **Prosody 분석 마지막** → 가장 까다로운 부분, 따로 개발 후 통합

> *쉬운 것부터 성공 경험 쌓기 → 백엔드 먼저, 앱 나중*

### 프론트엔드 방향
- 팀원 기술 스택에 따라 React(웹앱) / Flutter / React Native 중 선택
- **백엔드 파이프라인 완성 후 앱 연동**

## 회의 6 — 역할 분담 (5인 균등 분배)

### 최종 역할 분담

| 팀원 | 역할 | 주요 업무 | 디버깅 담당 |
|---|---|---|---|
| **A** | PM + STT | FastAPI 구조, Whisper 연동, 파이프라인 통합 주도 | STT 오인식 로그, 예외처리 |
| **B** | Prosody 분석 | librosa/Praat 피치 추출, 러시아 억양 모델링, 점수화 | 점수 이상치, 기준값 튜닝 기록 |
| **C** | AI 대화 + DTW | Gemini 프롬프트 엔지니어링, 시나리오 스크립트, DTW 유사도 | 영어 응답, 품질 개선 기록 |
| **D** | 프론트 (핵심) | 마이크 녹음, API 연동, 대화/메인 화면 | 마이크 권한, 포맷 불일치 |
| **E** | 프론트 보조 + QA | 시나리오·점수 화면, 버그 통합 관리, 파일럿 설계 | 전체 디버깅 문서 통합 |

### 역할 균등화 포인트
- 기존 D가 프론트 전체 담당 → **E가 프론트 일부 분담**
- 기존 C가 AI만 → **DTW 유사도 모듈 추가**
- E = 프론트 보조 + QA + 디버깅 총괄

## 회의 8 — 디버깅 강조 사항

### 교수님 강조: 디버깅 과정 문서화 필수

### 모듈별 예상 디버깅 항목

| 모듈 | 예상 오류 | 해결 방향 |
|---|---|---|
| STT | "배 아파요" → "배 아빠요" 오인식 | small 모델 업그레이드 or 후처리 |
| STT | 무음/짧은 음성 입력 시 오류 | 입력 전 길이 검증 추가 |
| Prosody | 잘 말했는데 20점 나옴 | 배경 잡음 전처리, 기준값 재조정 |
| AI | 영어로 대답 | 프롬프트에 "반드시 한국어로만" 명시 |
| AI | Gemini Rate Limit 오류 | 요청 간격 조절 (1초 딜레이) |
| 프론트 | 마이크 권한 거부 시 앱 멈춤 | 권한 거부 예외처리 추가 |
| 통합 | 각 모듈 따로는 되는데 연결 시 오류 | 데이터 포맷 통일 (API 명세서 기준) |

## 회의 9 — 음성 데이터셋 조사

### 필요한 데이터셋 종류
- **한국어 원어민 발화** → Prosody 비교 기준 레퍼런스
- **러시아어 억양이 섞인 한국어 발화** → 검증용 (공개 데이터셋 없음)

### 데이터셋 조사 결과

| 소스 | 내용 | 적합성 |
|---|---|---|
| AI Hub (aihub.or.kr) | 한국어 원어민 음성 데이터 | ✅ 레퍼런스용 적합 |
| Zeroth-Korean (OpenSLR) | 51.6시간 한국어 음성, CC BY 4.0 | ✅ 활용 가능 |
| Mozilla Common Voice | 다국어 오픈소스 음성 | △ 원어민 위주 |
| Golos (Sber, 러시아) | 1,240시간 러시아어 음성 | ❌ 순수 러시아어 |
| OpenSTT (러시아) | 4,000시간 러시아어 음성 | ❌ 순수 러시아어 |
| Kaggle Russian-English | 러시아-영어 이중언어 | ❌ 한국어 없음 |

### 결론
> 러시아어 억양 한국어 발화 공개 데이터셋은 세상에 존재하지 않음

### 최종 데이터 수집 전략
1. **AI Hub 한국어 원어민 데이터** → 레퍼런스 (Prosody 비교 기준)
2. **센터 직접 수집 (파일럿 테스트 겸)** → 고려인 아동 발화

### 직접 수집 방법
- 동의서 준비 (부모 동의서, 러시아어/한국어 병기)
- 녹음 스크립트: 시나리오 기반 문장 20~30개
- 수집 목표: 원어민 5명 + 고려인 아동 10명 × 문장 20개 = 약 300파일
- WAV 포맷, 16000Hz 또는 44100Hz
- 타임라인: W1~2 준비 → W9~10 파일럿 때 수집

## 주요 결정 사항 요약

### 프로젝트 핵심 방향
- **문제**: 고려인 아동은 한국어를 이해하지만 말할 기회가 없고, 러시아 억양이 굳어짐
- **해결**: AI가 한국어 말하기 상대 + 억양 시각적 피드백 + 게임형 앱
- **차별점**: 고려인 아동 특화 + 현장 검증 + Prosody 메인

### 기술 핵심 결정
- LLM: GPT-4o/Claude → **Google Gemini (무료 티어)**
- 앱: **Flutter** (팀원 역량 확정 후 React 웹앱으로 변경 가능)
- 백엔드: **FastAPI (Python)** — STT/Prosody/AI 모두 Python 기반

### 교수님 강조사항
- ✅ 디버깅 과정 문서화 필수
- ✅ 오류 케이스, 원인 분석, 수정 전/후 비교 기록
- ✅ 발표에서 디버깅 사례 제시
